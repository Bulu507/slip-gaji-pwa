# ARCHITECTURE RULES

## FEATURE MODULAR ARCHITECTURE

Each business domain must be implemented as a feature module.

Structure:

features/
domain-name/
application/
domain/
data/
presentation/

---

## LAYER RESPONSIBILITIES

presentation

UI components and pages.

application

Use cases and business workflows.

domain

Core models and domain logic.

data

Repositories and database access.

---

## DOMAIN ISOLATION

Each financial domain must remain independent.

Examples:

payroll
tunkin
uang-makan
lembur

Each domain maintains its own database stores.

Domains must not directly depend on other domains.

---

## IMPORT RULE

Every domain implements its own import service.

Workflow:

Upload
Parse
Preview
Validate
Commit

Import services must exist in the application layer.

---

## DATABASE RULE

Each domain uses its own IndexedDB stores.

Examples:

payroll_batches
payroll_transactions

Future modules:

tunkin_batches
tunkin_transactions

Domain stores must never be shared between modules.

---

## TRANSACTION RULE

Financial transactions are immutable.

Transactions cannot be edited.

Corrections must be performed by:

delete batch
reimport batch

---

## EMPLOYEE RULE

Employee module is not a master HR system.

Employee records are derived from consolidated payment data.

Employee module must never scan raw transaction stores.

Employees must be generated by the consolidation layer.

Employee attributes may be enriched without modifying
original transaction data.

---

## CONSOLIDATION RULE

Payment consolidation aggregates income from multiple modules.

Examples:

payroll
tunkin
uang makan
lembur

The consolidation engine produces derived datasets:

employee_payments
employees

Consolidation must never modify original transaction data.

---

## CONSOLIDATION LAYER

The consolidation layer exists outside feature modules.

Example structure:

core/
payment-consolidation/

Responsibilities:

aggregate transactions
build employee index
produce consolidated payments

This layer acts as the single source of truth for:

employee data
payment summaries
report generation

---

## SYNCHRONIZATION RULE

Synchronization must follow append-only model.

New batches are added.

Existing batches must never be modified.

---

## BACKUP RULE

Full database export must include all object stores.

Restore replaces entire local database.

---

## DOCUMENT GENERATORS

Document generation must consume aggregated payment data.

Documents must never read raw transaction stores.

Examples:

employee slip
monthly reports
annual reports

---

## CONSOLIDATION ENGINE TYPE

The payment consolidation engine uses
a materialized aggregation model.

Transactions
↓
Payment Consolidation
↓
employee_index
employee_payments

These datasets are persisted and used
as the primary data source for application features.

---

## CONSOLIDATION DATA TRACEABILITY

Consolidated payment records must keep references
to their originating import batches.

Example:

employee_payments.batchIds

This enables:

data traceability
batch level debugging
report validation
